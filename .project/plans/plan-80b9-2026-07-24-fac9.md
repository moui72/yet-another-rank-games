# Plan: Harden public-sharing clipboard copy

## Goal

Fix the missing error handling around the share-link clipboard copy so a rejected write surfaces to the user, and make the e2e coverage genuinely prove the copy succeeds.

## Scope

**In scope:**
- Wrap `copyShareLink`'s `navigator.clipboard.writeText` call in error handling and surface a visible failure state to the user (feedback F001).
- Grant clipboard permissions to the Playwright context in `e2e/sharing.spec.ts` so its copy-link assertion is a genuine regression check rather than a possible false positive (feedback F002).

**Out of scope:**
- Any change to the sharing feature's product behavior (visibility model, revocability, live-vs-snapshot) — all settled in `ui.md`/`datamodel.md`/`infrastructure.md` and unaffected here.
- Running the still-outstanding `npm run test:integration` / `npm run test:e2e` verification pass for the broader `public-list-sharing` feature (a pre-existing gap, not something this plan re-opens).

## Technical Approach

Both fixes are localized to code already written for `public-list-sharing`: `copyShareLink` in `src/routes/lists/[id]/+page.svelte` gets a try/catch around the clipboard write, flipping the UI's existing copy-status affordance to a failure message on rejection instead of silently no-op'ing. `e2e/sharing.spec.ts` gets `context.grantPermissions(['clipboard-read', 'clipboard-write'])` (Playwright's standard clipboard-permission API) added to its browser context setup, so the "Copied!" assertion is actually exercising a granted clipboard rather than one that may fail invisibly.
